# view-form-component — a view-form as a SANDBOXED WASM Component

This spike proves the enforcement layer behind **"no hallucinated UI: safe by
type + sandbox, not trust."**

A *view-form* turns a CEG attestation into a renderable 3D scene. If a view-form
can be **agent-generated**, it must be impossible for it to do anything *except*
return a scene — no kernel, no substrate, no network, no clock, no filesystem,
no WASI. This spike shows that a view-form compiled to a **WebAssembly Component
Model** component, against a WIT world that **imports nothing**, is
*structurally* incapable of side effects. The host runs it with an **empty
linker** (zero granted imports); the only data path out of the component is the
return value of a WIT-typed `render` function.

```
CEG `scores` envelope (JSON)
        │   (passed as a string across the WIT boundary)
        ▼
┌─────────────────────────────────────────────┐
│  view-form COMPONENT  (wasm32-unknown-unknown)│   imports: NONE
│  render(attestation-json) -> scene            │   exports: hello, render
│  pure · total · deterministic · no IO         │
└─────────────────────────────────────────────┘
        │   (typed `scene` returned by value)
        ▼
   wasmtime host, EMPTY Linker<()>  →  typed Scene IR  →  (→ out/scene.json → Bevy)
```

## What this proves (all green on this machine)

1. **A view-form is expressible as a zero-import component.** The built
   `view-form` component's own type declares **0 imports** and exactly 2 exports
   (`hello`, `render`). Verified two ways: `wasm-tools component wit` shows no
   `import` lines, and the host reads `component.component_type().imports(...)`
   and prints `imports granted-by-world: 0`.
2. **No WASI.** Built for `wasm32-unknown-unknown` (not wasip1/wasip2), the core
   module imports nothing — `wasm-tools print | grep '(import'` is empty — so
   there are no `wasi:*` capabilities to grant or deny in the first place.
3. **The host grants zero authority.** The component is instantiated against an
   `wasmtime::component::Linker::<()>::new(engine)` to which **nothing is ever
   added**, with store data `()`. There is no host state for the guest to reach.
4. **The typed Scene round-trips.** A real `scores` attestation
   (`dimension=evaluation:quality, score=0.85, confidence=0.92`) renders to a
   4-node Scene whose subject is a **green-dominant sphere** (score>0 ⇒ green),
   scaled by confidence — asserted in both the demo and the integration tests.
5. **An over-reaching component is rejected.** A `malicious` component that
   `import`s (and calls) a `host-caps/get-secret` capability **fails to
   instantiate** against the same empty linker:
   `component imports instance \`demo:malicious/host-caps@0.1.0\`, but a matching
   implementation was not found in the linker`. That failure *is* the
   enforcement.
6. **Totality.** Malformed/adversarial JSON never panics or traps; it yields a
   defined fallback scene (test `malformed_json_is_total_no_panic`).
7. **Two toolchain paths agree.** Path A (wit-bindgen + `wasm-tools component
   new`) and Path B (`cargo component build`) both produce zero-import
   components that the same host runs identically.

## The WIT world (`wit/view-form.wit`)

```wit
package demo:view-form@0.1.0;

world view-form {
  record vec3 { x: f32, y: f32, z: f32 }
  record rgba { r: f32, g: f32, b: f32, a: f32 }
  enum mesh-kind { sphere, cube, plane, cylinder }
  record scene-node {
    pos: vec3, scale: f32, mesh: mesh-kind, color: rgba,
    label: option<string>, source: string,
  }
  record camera { eye: vec3, look-at: vec3 }
  record scene { nodes: list<scene-node>, camera: camera, background: rgba }

  export hello: func(name: string) -> string;
  export render: func(attestation-json: string) -> scene;
}
```

The world has **only exports**. A guest built against it has no imports to
satisfy, so the sandbox is enforced by the *type of the world*, not by host
policy. The records mirror the locked-in Scene IR in
`../view-forms-spike/src/scene.rs` (the Bevy headless renderer's contract).

## Layout

```
view-form-component/
├── wit/view-form.wit         # the world (single source of truth)
├── guest/                    # PATH A: wit-bindgen guest -> core module
│   ├── src/lib.rs            #   the pure/total view-form: score→color, conf→scale, dim→label
│   └── wit/world.wit         #   copy of the world (bindgen path is manifest-relative)

│   ├── src/lib.rs
│   └── wit/world.wit
├── cc-guest/                 # PATH B: same logic via cargo-component (one-step)
│   ├── src/lib.rs            #   (src/bindings.rs is generated by cargo-component at build)
│   └── wit/world.wit
├── host/                     # wasmtime host: empty linker + typed call + sandbox proof
│   ├── src/lib.rs            #   bindgen! + render_with_empty_linker + import/export introspection
│   ├── src/main.rs           #   the demo (prints sandbox evidence + the typed scene)
│   ├── tests/scene_roundtrip.rs  # 4 integration tests (typed scene, totality, 0-imports, reject)
│   └── wit/view-form.wit
├── dist/                     # built components (view_form, malicious, cc_guest)
├── out/scene.json            # STRETCH: the returned Scene serialized for a JSON renderer
└── build.sh                  # builds both paths + runs tests + the demo, end to end
```

## Build & run

```bash
./build.sh
```

This (with `CARGO_TARGET_DIR=/tmp/wasm-component-target`):
1. builds the `guest` core module and lifts it to a component with
   `wasm-tools component new`, then prints its WIT (no `import` lines);
2. builds the `malicious-guest` component;
3. builds the `cc-guest` component via `cargo component build` (Path B);
4. runs `cargo test --release` in `host/` (4 tests);
5. runs the host demo against the view-form + malicious components, then against
   the cargo-component-built component.

Manual host invocation:

```bash
cd host
cargo run --release -- ../dist/view_form_component.wasm ../dist/malicious_component.wasm
```

### Why `wasm32-unknown-unknown` (not wasip1/wasip2)

`wasip2` emits a component but `std` references `wasi:cli/io/clocks/...`, so the
component declares WASI imports — **not** a closed sandbox. `wasip1` needs a WASI
adapter at `component new` time, which injects WASI imports. `wasm32-unknown-
unknown` has **no WASI at all**, so the verified result is **core-module imports
= NONE** and **component-world imports = NONE**. `serde_json` is pure compute, so
it runs there with zero host calls.

## The security boundary, precisely

- **What stops IO:** the WIT world grants no imports, so a conforming guest has
  no function to call to touch the outside world. The Component Model's type
  system makes "reach the host" un-nameable, not merely un-permitted.
- **What enforces it at runtime:** the host's `Linker<()>` is empty. Component
  instantiation is *total* over imports — any unsatisfied import is a hard
  instantiation error (demonstrated by the malicious component). A
  hallucinated/over-reaching view-form therefore **cannot even be loaded**, let
  alone run.
- **What the guest can return:** only a WIT-typed `scene`. The closed
  `mesh-kind` enum means there is no `Unknown`/escape variant; every node is
  exhaustively renderable. Bad input yields a defined fallback scene (totality),
  never a trap.

This is the structural backstop for agent-generated UI: a view-form is **safe by
type + sandbox**, so the rendering host never has to trust the agent that wrote
it.
